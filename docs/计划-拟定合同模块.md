# 契合「拟定合同」模块 · 最终搭建计划

> 本文档用于开工前的最终确认。确认无误后再开始编码。

---

## 一、这次到底做什么（范围）

只做**「拟定合同」这一条线**：用户用文字对话 → 拟定合同 → 预览 → 下载 Word / 获取模板。
搭一个轻量 Node 后端代理安全对接 Dify，前端采用 dc-runtime 驱动的单页应用（`app/index.html` + `web/index.html`）接通真实链路。

**明确不做（已确认）：**

- 文件上传（前端 UI + 后端接口都不做，属于「审查合同」，同学负责）。旧版 `_archive/legacy-v1/chat.html` 里现有的上传按钮保留成装饰性占位，不接逻辑。
- 后端不部署（先本地跑通）。
- 首页不加「合同模板」第三个快捷按钮。
- 账号体系、服务器端历史存储、Xcode 原生工程。

## 二、已敲定的关键决策

| 项目 | 结论 |
| --- | --- |
| 后端技术栈 | Node.js + Express |
| Dify 接入 | 官方云端，Chatflow（`advanced-chat`），一个聊天接口通吃所有意图 |
| 调试方式 | 先用 mock 假数据跑通，真实 Dify Key 由你之后自己填进 `.env` 验证（Key 不出现在对话里） |
| 交互 | SSE 流式打字机效果 |
| 对话持久化 | `localStorage` 存 `conversation_id` + 历史消息，刷新/重开仍在；从首页快捷标签进入 = 开新对话 |
| 终稿标记 | `<<<CONTRACT_START>>>...<<<CONTRACT_END>>>`（Dify 侧后续由对方接入，先按约定格式做检测） |
| 模板标记 | `<<<TEMPLATE:housing_lease>>>`（方案 B：模板是真实 Word 文件，存后端，Dify 只吐标识符） |
| Word 导出排版 | 精细排版（统一字体/页边距、标题层级、条款编号、签字栏区域） |
| 预览 + 下载 UI | 预览卡片用具体命名标题；下载区是独立文件卡片（左 Word 图标 + 文件名 + 大小，右圆形下载按钮），AI 终稿与模板复用同一组件 |

## 四、后端模块

1. **`difyClient.js`**：`sendChatMessage({ query, conversationId, user })` 调 Dify `/chat-messages`（`streaming`），SSE 流原样转发；mock 模式下返回一段带标记的假流（用来验证预览/下载链路，不依赖 Dify 真出标记）。
2. **`POST /api/chat`**：接 `{ query, conversationId, user }`，流式返回。
3. **`POST /api/export-docx`**：接合同终稿文字，用 `docx` 库生成精细排版 Word 返回下载。
4. **模板库三接口**：
   - `GET /api/templates`（列表）
   - `GET /api/templates/:id`（用 `mammoth` 把 docx 实时转预览）
   - `GET /api/templates/:id/download`（原文件下载）
   - 先放占位 `template.docx`，定稿后**只换文件、不改代码**。

## 五、前端架构（`app/index.html` + `web/index.html`）

> 注意：旧版双页面架构（`homepage.html` + `chat.html`）已归档至 `_archive/legacy-v1/`。
> 新版前端采用 dc-runtime 驱动的单页应用。App 移动端位于 `app/`，Web 桌面端位于 `web/`，两者共用同一套后端 API。
> API 桥接层 (`api-client.js`) 封装了 SSE 流式对话、合同导出、模板下载的完整逻辑。

改造要点：

1. 删除假回复逻辑（`replyFor()`），接真实 `/api/chat`。
2. API 地址自适应（`file://` 用线上地址，否则相对路径），兼容以后 WKWebView 套壳。
3. 匿名 `user` id 存 `localStorage`。
4. `conversation_id` + 历史消息持久化并恢复。
5. `fetch` + `ReadableStream` 流式渲染打字机效果。
6. 引入 `marked`（CDN）做 Markdown 渲染 + 基础转义。
7. 检测终稿/模板标记（缓冲拼接、隐藏标记原文），渲染「预览卡片 + 文件下载卡片」。
8. 流式进行中锁发送按钮，防重复。

## 七、自检出的风险点（已在方案中处理）

| 风险 | 处理方式 |
| --- | --- |
| 公开仓库 API Key 泄露 | Key 只放本地 `.env`，`.gitignore` 严格排除，仓库仅提交 `.env.example` 占位 |
| 流式输出把标记切碎 / 标记原文闪现 | 前端缓冲拼接后再匹配，展示时隐藏标记原文，闭合后才渲染卡片 |
| Dify 侧标记暂未上线 | mock 模式用带标记假流先跑通「检测 → 预览 → 下载」，对方上线后无缝生效 |
| SSE 被压缩/缓冲吞掉打字机效果 | SSE 路由单独设响应头、关压缩、及时 flush |
| Markdown 直插 innerHTML 的 XSS | 渲染时做基础转义 |
| 狂点发送导致会话串台 | 流式进行中锁发送按钮 |
| WKWebView 真机差异 | 用 `fetch`+`ReadableStream`（非 EventSource）；套壳注意事项写进 README |
